Executive Summary

Safepath is a single local-first desktop app implemented under File-organizerv2. In practice it is a Tauri 2 + React/Vite + Rust + SQLite application, not a client/server system. The real runtime path is: React UI -> Tauri IPC -> Rust commands -> safepath-core / safepath-store -> SQLite + filesystem.

The core workflow is materially implemented: source/destination selection, readiness probing, scan job creation, manifest paging, cheap and expensive duplicate analysis, protection detection and overrides, preset-driven planning, duplicate review, execution, history, undo, and learner suggestions with draft preset generation.

The weak side of the project is not the engine. It is documentation, onboarding, frontend maintainability, test breadth, roadmap traceability, and product polish.

Main Strengths

  • Rust domain logic is cleanly split into scanning, analysis, planning, execution, history, learner, rules, templates, and presets.
  • Persistence model is explicit and fairly complete, with SQLite tables for jobs, manifest, plans, sessions, action records, presets, and learner observations.
  • Execution is safety-oriented: preflight checks, cross-volume safe copy-then-remove, holding areas for duplicate cleanup, and undo support.
  • Rust unit test coverage is strong relative to project size.
  • Tauri DTO surface is typed end-to-end through src/types/app.ts and crates/safepath-core/src/types.rs.

Main Risks

  • src/App.tsx is a large monolith that owns nearly all frontend state, effects, polling, and workflow logic.
  • Docs are far behind the actual system; new contributors cannot learn the product from the repository alone.
  • No frontend or end-to-end automation means regressions in UI behavior and Tauri event wiring are easy to miss.
  • No CI means current healthy state is not protected.
  • Roadmap and original intent must be inferred from code and UI copy because formal planning artifacts are missing.

Architecture and System Audit

System model

The project is a single desktop application. No HTTP server, external API, queue service, cloud backend, or auth provider was found in first-party code. Storage is local SQLite plus direct filesystem operations. 

React UI (src/App.tsx)
  - invoke() and listen() through src/lib/tauri.ts
Tauri command layer (src-tauri/src/commands/*.rs)
  - delegates to safepath-core and safepath-store
Rust core logic (crates/safepath-core)
  - scan, analyze, plan, execute, undo, learner
SQLite store (crates/safepath-store)
  - jobs, manifest, analysis, plans, sessions, records, presets, learner observations
Local filesystem
  - source scanning, destination writes, duplicate holding, undo operations

Frontend

src/App.tsx controls almost everything: local inputs, selection sync, status hydration, scan polling, event listeners, manifest paging, plan review buckets, execution session refresh, history drill-down, undo flows, and learner interactions.

The three views are workflow, history, and settings. There is no router. Navigation is internal UI state.

Presentational components are small and focused:

  • src/components/layout/WorkflowShell.tsx: three-column layout
  • src/components/layout/AppStatusSummary.tsx: shell/platform/version/selection summary
  • src/components/permissions/PermissionReadinessCard.tsx: readiness state card

Tauri bridge

src/lib/tauri.ts defines all desktop commands and event listeners. The UI does not directly call Rust or filesystem APIs. If Tauri is unavailable, getAppStatus() returns a browser fallback DTO so the shell can render without crashing.

Implemented command surface includes status, path selection, presets, plan build/review, scan lifecycle, execution, undo, history, analysis, protection overrides, and learner endpoints.

Tauri shell and runtime state

src-tauri/src/main.rs creates an in-memory AppState that holds:

  • store: Store for SQLite persistence
  • scan cancellation tokens
  • selected source paths
  • selected destination paths
  • workflow phase

Important nuance: selected sources/destinations and workflow phase are not persisted to SQLite. They live only in app memory.

Filesystem permissions and readiness

src-tauri/src/permissions.rs does practical filesystem probing, not account-based permissioning:

  • sources are checked by metadata lookup and directory listing or file open
  • destinations are checked by creating and removing a temporary probe file
  • macOS-specific hints are returned for /Users/.../Library, /Volumes, and /Library

No authentication, roles, or admin permission system exists.

Core domain engine

Module Role
scanner.rs Walks source roots with walkdir, emits manifest-like records and progress.
analyzer.rs Computes category counts, structure signals, likely duplicates, definite duplicates, and protection detections.
rules.rs Evaluates planner rule conditions.
templates.rs Safely renders relative destination templates such as Images/{file_year}/{file_month}.
presets.rs Defines built-in presets: general_organize, project_safe, duplicate_review.
planner.rs Builds actionable plans from manifest + analysis + preset + destination root.
executor.rs Preflights plans, chooses execution strategy, performs file operations, records outcomes.
history.rs Builds history summaries and applies undo behavior.
learner.rs Turns user observations into learner suggestions and draft presets.
types.rs Shared DTO layer for frontend/backend serialization.

Persistence and schema

crates/safepath-store/src/migrations.rs defines schema version 2 and enables SQLite WAL mode. The database persists scan jobs, sources, manifest rows, analysis payloads, duplicate groups, protection overrides, presets, plans, execution sessions, action records, and learner observations.

Persistence is a hybrid JSON-plus-indexed-columns model. Most rich domain objects are serialized as JSON; some columns are broken out for indexing or migration support.

Data flow

  1. User enters source and destination paths in src/App.tsx.
  2. select_sources and select_destinations update in-memory selection state.
  3. start_scan creates a scan job and spawns a scan worker.
  4. record_manifest_entry persists each discovered entry and increments scan counters.
  5. analyze_manifest computes category counts, structure warnings, duplicates, and protection signals.
  6. build_plan loads manifest + analysis + preset and creates a PlanDto.
  7. Review actions and duplicate keeper choices mutate the plan and produce learner observations.
  8. execute_plan creates an execution session and appends action records as work progresses.
  9. get_history_page projects records into history entries with undo eligibility.
  10. Learner observations are aggregated into suggestions and draft preset previews.

Control flow

The practical control path is: readiness -> scan -> analysis -> plan -> review -> execute -> history/undo. Scan and execution use background worker threads; the frontend uses a mix of event listeners and polling to keep state updated.

User flow

  • Workflow: path setup, readiness, scan, analysis, planning, review, execute
  • History: record inspection, session inspection, best-effort undo
  • Settings: learner suggestions, draft previews, learner observations, suggestion feedback

Installation and Test Setup Guide

Required dependencies

  • Node.js and npm
  • Rust/Cargo
  • macOS desktop build prerequisites for Tauri builds

Exact Node and Rust versions are not pinned in-repo. This is a documentation and reproducibility gap, not a hidden requirement found in source.

Exact local setup steps

  1. Open the project root: File-organizerv2
  2. Install JS dependencies: npm install
  3. Run browser-only shell: npm run dev
  4. Run desktop dev app: npm run tauri:dev
  5. Run lint: npm run lint
  6. Run production build: npm run build
  7. Run Rust tests from this checkout: CARGO_HOME="$PWD/.cargo-home" cargo test --workspace --offline
  8. Run production-like desktop packaging: npm run tauri:build

Common setup issues and fixes

  • Port 1420 in use: Vite uses strictPort: true in vite.config.ts. Free the port or change config.
  • macOS privacy blocks path access: avoid protected folders or grant the needed OS-level permissions.
  • Rust tests fail when Cargo writes to default home: use the repo-local .cargo-home command shown above.
  • Misleading docs: do not rely on the current README.md; it is still a generic Vite template.

Missing items that prevent reliable local testing

  • No real setup guide
  • No version pinning
  • No fixture corpus
  • No frontend tests
  • No E2E tests
  • No CI
  • No single verification script that runs all important checks

Full Usage and Operations Guide

User-facing features

  1. Enter one absolute source path per line in the workflow left panel.
  2. Enter the primary destination folder in the right panel.
  3. Click Check readiness to verify source readability and destination writability.
  4. Click Start scan to create a scan job and traverse the selected source trees.
  5. Review manifest pages and analysis summary.
  6. Optionally click Run expensive analysis to hash files and improve duplicate detection certainty.
  7. Select a preset and click Build plan.
  8. Approve, reject, or reset planned actions. For duplicate groups, choose a keeper first.
  9. Optionally mark detected paths as protected.
  10. Click Execute approved.
  11. Use the History tab to inspect records and run record-level or session-level undo.
  12. Use the Settings tab to inspect learner suggestions and save draft presets.

Developer/operator workflows

  • Review built-in presets in crates/safepath-core/src/presets.rs.
  • Inspect runtime schema in crates/safepath-store/src/migrations.rs.
  • Use Rust tests as the main automated safety net today.
  • Inspect the runtime DB at the Tauri app data location if debugging persisted state.
  • Use history/undo as the only built-in operational recovery mechanism.

Admin/maintenance tasks

There is no real admin console. Maintenance today is mostly:

  • running lint/build/tests
  • inspecting the SQLite database
  • reviewing history and undoing records/sessions
  • adjusting presets through code or learner-generated draft presets

Troubleshooting guide

  • Scan will not start: inspect readiness summary and verify destination exists.
  • Duplicate group approval fails: pick a keeper first.
  • Execution shows preflight issues: inspect missing source, destination collision, or missing keeper constraints.
  • Unexpected protected path behavior: inspect analysis protections and user overrides.
  • Undo unavailable: check whether the action was completed, rollback-safe, and not already undone.

Original Plan vs Current Implementation

Formal planning artifacts are largely absent. There is no meaningful product README, no roadmap, no ADRs, and no project Git history in this checkout. As a result, original intent must be reconstructed from product copy and code-level signals.

Best evidence for intended direction

  • Rename in progress: product is Safepath in config and UI, but legacy file-organizerv2 still appears in HTML and lockfile metadata.
  • Sprint 1 single-destination: UI explicitly says only one destination root is used today, while backend still stores a list.
  • Phase 6 undo: undo-safe strategy language suggests milestone naming that outlived the implementation.
  • Phase 7 learner: settings copy frames the learner as a later milestone.
  • Future preset-level duplicate keeper policy: learner suggestion text explicitly points to future policy work.

Recommendations and Next Steps

Must-fix items

  • Replace the stock README.md with real product documentation for Safepath.
  • Clean up naming drift across README.md, index.html, and package-lock.json.
  • Add CI for npm run lint, npm run build, and cargo test --workspace.
  • Modularize src/App.tsx into feature-level components/hooks.
  • Decide whether repo-local .cargo-home is intentional and document or ignore it accordingly.

Quick wins

  • Add .nvmrc and rust-toolchain.toml.
  • Add a single verification command or script that runs lint, build, and Rust tests.
  • Wire job_failed and analysis_progress into the UI.
  • Update settings copy so it matches current learner draft-save behavior.
  • Document that only the first destination root is used today.

Critical blockers

  • No real onboarding docs
  • No CI
  • No frontend or integration test coverage
  • No backup/snapshot safety net beyond best-effort undo
  • No formal planning/spec trail

Medium-term improvements

  • Add frontend smoke tests and one desktop end-to-end happy-path test.
  • Persist path selections across app restarts if that matches intended UX.
  • Move expensive analysis into a background worker with progress events.
  • Add richer filtering and search across manifest, plan, and history.
  • Build a real preset management UI.
  • Improve structured logging and error visibility.

Long-term architecture / product recommendations

  • Decide whether Safepath remains a narrow local organizer or grows into a broader organization platform.
  • If it stays local-first, prioritize snapshots/backups and stronger undo semantics before expanding breadth.
  • If product scope expands, prioritize search, import/export, tagging, and accessibility before telemetry or notifications.
  • If multi-device use is ever intended, sync and conflict resolution will require a substantial architectural step beyond the current local-only design.

Optional enhancements

  • Search across manifest, history, actions, and presets
  • Preset import/export
  • Better duplicate review UX with metadata previews instead of entry IDs
  • Notifications for long-running scans and execution sessions
  • Analytics only if accompanied by an explicit privacy and product strategy

Closing Assessment

Safepath is not just a scaffold anymore. The engine and persistence layers are real, the end-to-end workflow is implemented, and the system can be exercised locally today. The main gap is that the surrounding product layer has not caught up with the code. A focused iteration on docs, CI, frontend modularity, and small UX/runtime cleanup would materially improve confidence and maintainability.
